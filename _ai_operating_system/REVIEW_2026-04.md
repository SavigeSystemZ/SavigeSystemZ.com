## P0 — CRITICAL

### 1. Security

#### Node.js Crypto in Runtime Context (Potentially Edge-unsafe)
- **Finding:** `lib/auth.ts:2` imports `createHmac, randomBytes, timingSafeEqual` from `node:crypto`. While this is safe in the **server runtime** (not Edge), it violates platform-agnostic design.
  - `lib/signed-download.ts:1` also uses `node:crypto` (server-safe).
  - `lib/vault-crypto.ts:1` uses `node:crypto` (server-safe but creates cipher overhead).
- **Impact:** If these functions are ever imported into an Edge runtime (middleware, proxy), they will fail.
- **Recommendation:** Audit all `node:crypto` imports for actual Edge usage. Migrate auth-critical functions to Web Crypto (`lib/hmac-web.ts` pattern). Vault encryption is server-only, so acceptable; add a comment.

#### Default/Weak Secrets in Proxy and Auth
- **Finding:** `proxy.ts:8` and `lib/auth.ts:15` both read `OWNER_LOGIN_SECRET ?? "change-me-in-production"`.
- **Impact:** In development, plaintext default secret is a vector; in production, a misconfigured secret breaks admin security.
- **Recommendation:** Add a startup check: if `NODE_ENV === "production"` and secret is unset or weak, fail immediately with a clear error. `proxy.ts:30-34` already does this partially; extend to `lib/auth.ts`.

#### Idempotency & Replay Protection Missing from Stripe Webhook
- **Finding:** `app/api/webhooks/stripe/route.ts` validates signature but doesn't deduplicate by `event.id`.
- **Location:** `lib/stripe-webhook-processor.ts:8` processes `checkout.session.completed` once per webhook call.
- **Risk:** If Stripe retries the same webhook (network glitch), `completePurchaseFromSessionId` may charge twice.
- **Recommendation:** Before `processStripeWebhookEvent`, check if `event.id` already exists in a `StripeWebhookLog` table or Redis. Return 200 immediately if already processed (idempotent).

#### Vault Mutation Rate Limit Missing Per-User Scoping
- **Finding:** `lib/vault-rate-limit.ts:26` rate-limits by IP only (`vault:mut:${ip}`).
- **Impact:** On shared networks (corporate WiFi, cloud CI), one user's abuse blocks all users on that IP.
- **Recommendation:** If auth context is available, use `userId` as primary key; fall back to IP for anonymous. Adjust thresholds accordingly.

#### No CSRF Token on State-Changing Forms
- **Finding:** Public endpoints like `/api/project-requests`, `/api/submissions`, `/api/checkout` accept POST but don't validate CSRF tokens.
- **Impact:** Low-risk for API-driven flows, but if any HTML form directly POSTs to these endpoints, CSRF applies.
- **Recommendation:** Verify all POST routes that serve to browser clients use SameSite=Lax cookies (already set in `checkout/complete/route.ts:33`). Add a hidden `x-csrf-token` to any HTML forms that POST.

- **Recommendation:** Document the cascade strategy in schema comments. Consider cascading Application deletes to ApplicationVersion, Media, License, Purchase, etc. Or enforce explicit deletion logic in app code.

## P1 — QUALITY & PRE-LAUNCH POLISH

### 1. Security

#### Admin PATCH/POST Routes Missing Idempotency Keys
- **Finding:** `app/api/admin/code/[id]/route.ts:PATCH` (line 13–70) and `:POST` (line 79–98) accept requests but don't deduplicate retries.
- **Impact:** Double-clicking "sync" button could run sync twice, causing redundant GitHub API calls and audit spam.
- **Recommendation:** Add an `idempotencyKey` field to request body or use request header `Idempotency-Key`. Store in cache/DB; return cached result on retry.

#### No Rate Limit on Admin Routes
- **Finding:** Admin routes like `POST /api/admin/code/[id]` (sync), `PATCH /api/admin/applications/[id]` have no rate limiting.
- **Impact:** Compromised owner account could DoS backend (GitHub API spam, DB writes).
- **Recommendation:** Apply `rateLimit()` to mutation routes, keyed by user ID. Example: `rateLimit(context.userId, 30, 60_000)` for 30 ops/min per owner.

#### Missing Validation on Admin JSON Uploads
- **Finding:** `app/api/admin/versions/route.ts`, `app/api/admin/release-assets/route.ts` call `.json()` but no JSON size limit.
- **Impact:** Malicious requests could send 100MB JSON, consuming memory.
- **Recommendation:** Wrap `request.json()` in a custom parser with size limit (e.g., 1MB). Use `request.text()` with length check first.

#### `getAuthContext()` Doesn't Validate Session Expiry on Every Call
- **Finding:** `lib/auth.ts:55–61` checks `session.expiresAt.getTime() > Date.now()` but doesn't auto-delete expired sessions.
- **Impact:** Database grows with stale sessions; memory waste.
- **Recommendation:** Add a background job (cron) to delete sessions where `expiresAt < now()` daily.

#### No Signed-Download Token TTL Validation
- **Finding:** `lib/signed-download.ts:38` checks `parsed.exp < Math.floor(Date.now() / 1000)` but tokens can be valid for arbitrarily long.
- **Example:** `app/api/files/signed/route.ts` doesn't set a TTL on the token; it relies on the caller.
- **Recommendation:** Enforce a max TTL (e.g., 1 hour) when creating tokens. Reject tokens older than that, regardless of `exp` field.

### 2. Data Layer

#### Missing Unique Constraint on Purchase.stripeCheckoutSessionId + Email
- **Finding:** `Purchase.stripeCheckoutSessionId` is `@unique` (line 168), but Stripe webhooks could be processed twice by different instances if not idempotent.
- **Recommendation:** Already handled by unique constraint. Ensure webhook dedup still applies (see P0 idempotency issue).

#### No Isolation Level on Transaction-Critical Paths
- **Finding:** `lib/checkout-complete.ts:63–73` creates License and updates Purchase in sequence but not in a transaction.
- **Risk:** If second write fails, License is granted but Purchase not marked COMPLETED.
- **Recommendation:** Wrap in `db.$transaction([...])`.

### 3. Commerce

#### No Refund Flow
- **Finding:** Schema has `PurchaseStatus` enum (PENDING, COMPLETED, FAILED) but no REFUNDED state or refund handling.
- **Impact:** If a customer requests refund, no way to track it.
- **Recommendation:** Add REFUNDED status. Create `POST /api/admin/purchases/[id]/refund` endpoint with Stripe refund call + audit log.

#### Checkout Mock Mode Uses Predictable Session IDs
- **Finding:** `app/api/checkout/route.ts:24` generates mock session as `cs_mock_${randomBytes(16).toString("hex")}`.
- **Impact:** Not a security risk (dev-only), but weak for testing. Could collide if two users checkout simultaneously.
- **Recommendation:** Use `randomUUID()` instead for better distribution.

#### Missing Stripe Event Type Dispatch
- **Finding:** `lib/stripe-webhook-processor.ts:8` only handles `checkout.session.completed`. No handling for `invoice.payment_failed`, `customer.subscription.deleted`, etc.
- **Impact:** Failed payments are silently ignored.
- **Recommendation:** Add handlers for at least `payment_intent.payment_failed` to mark Purchase as FAILED.

### 4. Storage

#### S3 Presign TTL Not Enforced at Upload
- **Finding:** `lib/s3-release-presign.ts:5` uses `DEFAULT_PUT_EXPIRES = 900` (15 min). ✓ Good.
- **Finding:** `lib/s3-presign.ts:4` uses `DEFAULT_EXPIRES = 300` (5 min). ✓ Good.
- **Finding:** Callers can override with `params.expiresInSeconds`. No cap.
- **Recommendation:** Add a max: `Math.min(params.expiresInSeconds ?? DEFAULT_EXPIRES, 3600)` to prevent 24h presigns.

#### No Content-Type Validation on Upload URLs
- **Finding:** `lib/s3-release-presign.ts:68` accepts any `contentType` from request and passes to S3.
- **Risk:** Client could upload `.exe` to `releases/` bucket.
- **Recommendation:** Whitelist allowed MIME types: `application/gzip`, `application/x-zip`, `text/plain`, `application/json`, `application/octet-stream`. Reject others.

#### No S3 Vault Scan Lambda Hook Wired
- **Finding:** Schema has `VaultArtifact`, `s3-vault-presign.ts` exists, but no integration with `infra/s3-vault-scan-lambda/` for antivirus/YARA scanning.
- **Impact:** Users could upload malware to vault.
- **Recommendation:** Add S3 event notification from vault bucket to Lambda. Scan uploaded files; quarantine if infected.

### 6. AI/Agentic

#### Prompt Injection Not Hardened
- **Finding:** `packages/ai/src/index.ts:6–8` sanitizes input by stripping `<>` only.
- **Risk:** Attacker can inject text to confuse the knowledge base search or override context.
- **Recommendation:** Use a more robust sanitizer: strip special characters, limit length to 500 chars, reject common injection patterns (e.g., "ignore previous", "system prompt").

#### No Rate Limit Keyed by User/Session
- **Finding:** `app/api/ai/chat/route.ts:9` uses generic key `"ai-chat"` for all users.
- **Impact:** One malicious user can exhaust quota for all users.
- **Recommendation:** Extract user ID from context or use IP + session. Implement per-user limit (e.g., 60 req/min).

#### AI Response Not Audited
- **Finding:** No audit log when `/api/ai/chat` is called.
- **Impact:** Can't track abuse or user intent.
- **Recommendation:** Log to `AuditLog`: action="ai_chat.request", metadata={query, topic, userAgent}.

### 1. Security

#### Session Token Rotation Not Implemented
- **Finding:** Sessions are created once and expire after 12 hours. No rotation.
- **Recommendation:** For high-security workflows, rotate token on each request and store last 2 tokens. Low priority.

#### No Audit Log Retention Policy
- **Finding:** `AuditLog` has no `deletedAt` or TTL.
- **Impact:** Audit logs grow unbounded.

## Notes

- **Code Quality:** Codebase is well-structured, follows TypeScript strict mode, uses Zod validation consistently, and has reasonable test coverage. Security posture is strong but needs idempotency and rate-limit hardening pre-launch.
- **Architecture:** Clean separation of concerns (API routes, lib utilities, components). Prisma + PostgreSQL is solid. Next.js App Router + Turbopack is modern.
- **Risk Profile:** Medium-risk (e-commerce, user auth, vault encryption). No critical CVEs spotted. Main risks are idempotency (commerce) and vault DoS (rate limit).
